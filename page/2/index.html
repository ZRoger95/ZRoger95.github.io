什么是Memcached ?

JSON Web Token

传统的认证

​ 在以前的单体web应用中，由于 http 协议是一个无状态的协议，当客户端发送请求时，并不知道是哪位用户发出的请求。为了解决这个问题，在用户登录后，服务器会存储一份用户登录的信息（session)，这份登录信息会在响应时传递给浏览器，保存在cookie中，以便下次发生请求时传递给应用。

遭遇的问题

​ 随着用户量的增多，服务器的开销也开始增加，登录用户们的信息都被存储在当前这台服务器的内存中。限制了分布式的扩展，限制了负载均衡的能力。使用cookie容易受到CSRF

并发场景

同时请求时，过期问题

JWT的缺陷主要是在并发的时候体现的，要理解这个问题首先你得知道一些基本特点：

  • JWT的刷新操作也就是返回一个新的JWT(这是由于JWT的 过期时间exp claim是封装在Payload中的)
  • 在实际的使用中，刷新后旧的Token会被放入黑名单，避免被再次使用，导致重放攻击。
  • 我们一般是在一个业务HTTP请求的同时刷新JWT，附带在响应中返回，尽量的保证在用户使用期间内，JWT不过期。

当你清晰的理解上面的特点后，你会发现：

如果有两个并发请求，附带着相同的Token请求服务器。此时一定会有一个先后顺序，如果其中的一个请求的Token先刷新，那么Token被放入黑名单，无法使用，后来的请求就被服务器拒绝。

这的的确确是存在的问题，而导致问题的根本原因就是：我们在每一次请求中都进行了JWT刷新，旧Token放入黑名单。

方法 Method

Go 中虽然没有class，但依旧有method，通过显示说明receiver来实现与某个类型的组合，只能为同一个包中的类型定义方法，receiver可以是类型的值或指针。

method不存在方法重载，可以是使用值或指针来调节方法，编译器会自动完成转换，从某种意义上来说，方法是函数的语法糖，因为receiver其实就是方法所接收的第一个参数（Method Value vs Method Expression）

如果外部结构和嵌入结构存在同名方法，则优先调用外部结构的方法，类型别名不会拥有底层类型所附带的方法。

方法可以待用结构中的非公开字段

接口 interface

接口是一个或多个方法签名的集合，只要某个类型拥有该接口的所有方法签名，即算实现该接口，无需显示声明了实现了哪个接口，这称为 Structural Typing

接口只有方法声明，没有实现，没有数据类型

接口可以匿名嵌入其他接口，或嵌入结构中

将对象赋值给接口时，会发生拷贝，而接口内部存储的是指向这个复制品的指针，即无法修改复制品的状态 ，也无法获取指针

只有当接口存储的类型和对象都为 nil 时，接口才等于 nil

接口调用不会做 receiver 的自动转换

接口同样支持匿名字段方法

接口可实现类型 OOP 中的多态

空接口可以作为任何类型数据的容器

结构

Go 中的struct与 C 中的 struct非常相似，并且 Go 没有 class 。使用 type<Name> struct{}定义结构，名称遵循可见性规则。

  • 支持指向自身的指针类型成员
  • 支持匿名结构，可用作成员或定义成员变量
  • 匿名结构也可以用于map的值
  • 可以使用字面值对结构进行初始化
  • 运行直接通过指针来读写结构成员
  • 相同类型的成员可进行直接拷贝赋值
  • 支持==与!=比较运算符，但是不支持>和<
  • 支持匿名字段，本质上是定于了以某个类型名为名称的字段
  • 嵌入结构作为匿名字段看起来像继承，但不是继承
  • 可使用匿名字段指针
