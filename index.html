Zuul Filter

目前项目中用到的是 Zuul core 是 1.x 版本，可以继续尝试使用Zuul 2，或者用GateWay替换，当然还有其它一些网关中间件可以替换 Zuul , 这里只是一个备注

【译】编写Filter

原文

Filter 基础

Filter 接口

自定义Filter需要继承 ZuulFilter ，并且实现其中的方法

运行或者停止

shouldFilter返回一个boolean值来决定Filter是否运行

优先级

filterOrder方法返回int值来决定过滤器之间的运行顺序

Filter Types

可以用任何的字符串来定义Filter的类型，主要由2种用途：

  1. Zuul 中基本的请求生命周期，按照顺序由：pre、routing和post阶段构成。所有这些类型的Filter都会运行在所有的请求中
  2. 任何类型的Filter都可以使用GroovyProcessor.runFilters(String type))方法显示运行

Filter 协调

Filter没有直接方法去访问彼此。可以使用RequestContext共享状态，RequestContext是一个类似Map的结构，具有一些显示访问器方法，来获取Zuul原始数据。

特殊Filter扩展

StaticResponseFilter

StaticResponseFilter允许从Zuul本身生成响应，而不是将请求妆发到源

SurgicalDebugFilter

SurgicalDebugFilter允许将特定请求路由到单独的调试集群或主机中

了解了以上，再来处理一些与过滤器有关的设计，比如：

  1. 设计双因素认证拦截
  2. 用户登录后成功签发bearer token
  3. refresh token 自动续签问题
  4. 黑白名单拦截设计

其他的补充

  • PRE Filter 在请求到源之前执行。比如：请求身份认证，选择源服务和记录调试信息
  • ROUTING Filter 处理请求路由到源。这是使用 Apache HttpClient 或是 Netfix Ribbon 构建和发送原始HTTP请求的地方
  • POST Filter 在请求被路由到源后执行。比如：向响应头加入标准HTTP标头，收集统计信息和指标以及将响应从源传输到客户端
  • ERROR Filter 在其他阶段之一发送错误时执行

除了默认的过滤器流程，Zuul 允许我们创建自定义过滤器类型并显示执行它们。

JWT的工作过程

客户端接收服务器返回的JWT，将其存储在Cookie或localStorage中。

此后，客户端将在与服务器交互中都会带JWT。如果将它存储在Cookie中，就可以自动发送，但是不会跨域，因此一般是将它放入HTTP请求的Header Authorization字段中。

当跨域时，也可以将JWT被放置于POST请求的数据主体中。

Git的简单回滚操作

就算Git中分支、提交、gitflow 分的都很清楚，也难免会遇到需要回退到前几个版本的尴尬境地，你都是怎么做的呢？施展CV大法吗？😂
两个简单的命令就可以让他滚回去：

我们来试试查看最近的三个提交是怎么样的 git log -3

git log -3

我们可以看到每一个提交从上到下都有，对应的哈希值、作者、邮箱、日期，以及我上面提到的提交信息，非常重要，不然你都不知道这次提交是为了什么，甚至找不到滚回去的位置。

假设我们要回滚到第三条，也就是5d9c3b1dab8de8932ebf8deffef821cf2052fd65 这一条，执行命令git reset --hard 5d9c3b1dab8de8932ebf8deffef821cf2052fd65 即可，当前分支就回退到这个版本了。

回退后，我们可能会修改代码，这个时候又有新的问题了，以前的提交已经push到远程仓库，也就是我们目前的代码落后两个版本，同时有一个新的commit待提交。
两个方式：
1、强势一点：硬提
git push -f origin [分支名称]
2、委婉的：先拉去->解决冲突->重新提交整合后的代码
git pull origin develop -> solve conflct -> git push -u origin develop
记得写清楚提交信息鸭！！！！！

这样一个简单的回滚就完成了。
不要告诉我你不会解决冲突。🤓🤓🤓🤓🤓
