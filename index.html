Spring Cloud

前言

总结工作到现在的使用Java Web框架的变化 ( 2017 ~ 2020 )

  1. 最开始接触 Java Web 的时候，那时候用的框架是 SSH 既 Struts2 + Spring + Hibernate。围绕着struts作为系统的基础架构，负责 MVC 的分离，struts 负责模型部分，hibernate 负责数据交互（持久层），spring 负责管理配置。

  2. 之后一个阶段，抛弃了struts2，转成了 SSM : Spring-MVC + Spring + MyBatis，或者是 SSH : Spring-MVC+Spring+Hibernate。

不同点在于对控制视图（Views）和模型的交互规制不同，Struts2 是 Action ，SpringMVC 是方法级别，便于实现RESTful风格（前后端分离）。 主要在于 MVC 的实现方式，以及 ORM 持久方面的不同。SSM优点在注解开发上，且ORM实现更加灵活。SSH注重配置开发。

  1. 最后到了现在的Spring Cloud + Spring Boot + Mybatis。也不是所有的场景都需要分布式的微服务框架。如果场景单一，使用Spring Boot + Mybatis 或者 Spring Boot + Hibernate 来开发并维护单应用的项目。随着用户量逐渐增多，每日请求也开始变多，并发的问题，频繁更新所带来的问题等等。需要多维度解决如上发生的问题，
    1. 比如电商的场景，将用户登录，订单，售后，秒杀这些抽离出来，即使其中一个服务出现了问题，也不会影响到其他服务。
    2. 比如项目是一些产品的组合，有的客户需要A+B，有的客户需要A+B+C，在开发时，根据产品的粒度，区分开A，B，C三种产品，这些是业务层的服务。它们还用共同用了无业务类型的微服务，如：身份验证（auth），邮件短信平台，文件存储服务等等

敏捷开发，微服务这些概念的提出都离不开 Martin Fowler

定义

Spring Cloud是一站式微服务解决方案，它像是一个桌子，由你把各个积木（微服务）搭配起来，展示最终的效果。和Spring Cloud差不多，还有阿里的Dubbo。或者是Google的GPRC

文档： https://start.spring.io/actuator/info

Spring Cloud主要解决了一些纬度的问题

思路

  • 数组下标
  • 快慢指针

使用快慢指针来记录遍历的坐标。

  • 开始时这两个指针都指向第一个数字
  • 如果两个指针指的数字相同，则快指针向前走一步
  • 如果不同，则两个指针都向前走一步（之后将慢指针所在的节点值换成快指针所指位置）
  • 当快指针走完整个数组后，慢指针当前的坐标加1就是数组中不同数字的个数

Plan 9

Plan 9是什么

它是一个网络操作系统（大规模集群）或者说是一种概念，来自贝尔实验室，由Ken Thompson、Dennis Ritchied等大神开发。

是一个Unix系统，想真正的实现一切皆文件这个想法。Google中的绝大部分资料，从各种存储系统到RPC服务，都可以用类似的/gfs/…/bigtable/…的文件名方式在全部Google集群中的唯一寻址，很大部分存储系统统一使用FIle API系统访问。（/proc文件系统概念还有UTF-8）（/net/tcp文件系统）

这个可能是它的官网：http://9p.io/plan9/

Plan 9与Go

Go语言的实现使用到了Plan 9的汇编器，与AT&T和Intel的汇编器不同，也不是C语言内嵌汇编的语法。了解Go的汇编是为了学习Go的实现细节
