JSON Web Token

传统的认证

​ 在以前的单体web应用中，由于 http 协议是一个无状态的协议，当客户端发送请求时，并不知道是哪位用户发出的请求。为了解决这个问题，在用户登录后，服务器会存储一份用户登录的信息（session)，这份登录信息会在响应时传递给浏览器，保存在cookie中，以便下次发生请求时传递给应用。

遭遇的问题

​ 随着用户量的增多，服务器的开销也开始增加，登录用户们的信息都被存储在当前这台服务器的内存中。限制了分布式的扩展，限制了负载均衡的能力。使用cookie容易受到CSRF

字段 说明
iss jwt签发者

aud 接收jwt的一方
exp jwt的过期时间，这个过期时间必须要大于签发时间

公共声明

可以添加任何信息，如用户信息或重要业务信息，但不建议添加敏感信息，该部分客户端可解密

私有声明

由提供者和消费者共同定义的声明，一般不建议存放敏感信息。

并发场景

同时请求时，过期问题

JWT的缺陷主要是在并发的时候体现的，要理解这个问题首先你得知道一些基本特点：

  • JWT的刷新操作也就是返回一个新的JWT(这是由于JWT的 过期时间exp claim是封装在Payload中的)
  • 在实际的使用中，刷新后旧的Token会被放入黑名单，避免被再次使用，导致重放攻击。
  • 我们一般是在一个业务HTTP请求的同时刷新JWT，附带在响应中返回，尽量的保证在用户使用期间内，JWT不过期。

当你清晰的理解上面的特点后，你会发现：

如果有两个并发请求，附带着相同的Token请求服务器。此时一定会有一个先后顺序，如果其中的一个请求的Token先刷新，那么Token被放入黑名单，无法使用，后来的请求就被服务器拒绝。

这的的确确是存在的问题，而导致问题的根本原因就是：我们在每一次请求中都进行了JWT刷新，旧Token放入黑名单。
