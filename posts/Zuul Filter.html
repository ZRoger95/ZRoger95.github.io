原文

Filter 基础

Filter 接口

自定义Filter需要继承 ZuulFilter ，并且实现其中的方法

运行或者停止

shouldFilter返回一个boolean值来决定Filter是否运行

优先级

filterOrder方法返回int值来决定过滤器之间的运行顺序

Filter Types

可以用任何的字符串来定义Filter的类型，主要由2种用途：

  1. Zuul 中基本的请求生命周期，按照顺序由：pre、routing和post阶段构成。所有这些类型的Filter都会运行在所有的请求中
  2. 任何类型的Filter都可以使用GroovyProcessor.runFilters(String type))方法显示运行

Filter 协调

Filter没有直接方法去访问彼此。可以使用RequestContext共享状态，RequestContext是一个类似Map的结构，具有一些显示访问器方法，来获取Zuul原始数据。

特殊Filter扩展

StaticResponseFilter

StaticResponseFilter允许从Zuul本身生成响应，而不是将请求妆发到源

SurgicalDebugFilter

SurgicalDebugFilter允许将特定请求路由到单独的调试集群或主机中

了解了以上，再来处理一些与过滤器有关的设计，比如：

  1. 设计双因素认证拦截
  2. 用户登录后成功签发bearer token
  3. refresh token 自动续签问题
  4. 黑白名单拦截设计

其他的补充

  • PRE Filter 在请求到源之前执行。比如：请求身份认证，选择源服务和记录调试信息
  • ROUTING Filter 处理请求路由到源。这是使用 Apache HttpClient 或是 Netfix Ribbon 构建和发送原始HTTP请求的地方
  • POST Filter 在请求被路由到源后执行。比如：向响应头加入标准HTTP标头，收集统计信息和指标以及将响应从源传输到客户端
  • ERROR Filter 在其他阶段之一发送错误时执行

除了默认的过滤器流程，Zuul 允许我们创建自定义过滤器类型并显示执行它们。
